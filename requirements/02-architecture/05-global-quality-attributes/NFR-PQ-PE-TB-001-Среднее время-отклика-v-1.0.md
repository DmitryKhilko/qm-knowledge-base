**ID:** NFR-PQ-PE-TB-001  
**Наименование:** глобальный атрибут качества "Среднее время отклика"  
**Атрибут качества:** Performance Efficiency  
**Податрибут:** Time behavior  
**Приоритет:** High  
**Ответственный:** Дмитрий Хилько  
**Инициатор:** Виталий Тугай  
**Статус:** Одобрен  
**Версия:** 1.0  

**Описание:** Среднее время отклика системы на запросы пользователей при стандартной нагрузке должно быть не более 2 секунд. 
Измеряется на базе 95% запросов, выполненных за этот период.  

**Обоснование:** Снижение времени отклика критично для улучшения пользовательского опыта и поддержания высокой кон-версии.  

**Область действия:** Все запросы, выполняемые через основной API интерфейс, включая запросы к БД и обработку на сервере.  

**Исключения:** Не учитываются запросы при аварийном режиме системы или во время плановых технических работ.  

**Допущения:** Средняя нагрузка системы составляет до 1000 RPS (Requests Per Second) на весь рабочий день.  

**Шкала измерения:** Time  
**Единица измерения:** milliseconds  

**Метод измерения:** Среднее время от получения HTTP-запроса до отправки ответа, включая обработку на сервере и запросы к базе данных.  

**Формула:** AverageResponseTime = Σ(ResponseTime) / N, где N — количество обработанных запросов.  

**Тестовый метод:** Нагрузочное тестирование с использованием JMeter для симуляции реальных запросов пользователей в усло-виях нормальной и повышенной нагрузки.  

**Уровни:**  
**Текущий:** 1800 ms (среднее время отклика за последние 6 месяцев)  
**Минимально допустимый:** ≤ 2000 ms (для 95% запросов)  
**Целевой:** ≤ 1500 ms (для 95% запросов)  
**Идеальный:** ≤ 1000 ms (для 95% запросов)  

**Сценарий:** Запросы пользователей, не превышающие установленного лимита по нагрузке.  

**Триггер:** Пользователь отправляет запрос через API, например, для оформления заказа или получения информации.  

**Когда:** В рабочие часы системы (08:00–18:00, Понедельник–Пятница).  

**Где:** Производственная среда, нагрузка на сервер соответствует реальному поведению пользователей.  

**Риск:**  Падение производительности при увеличении нагрузки или при высокой частоте запросов.

**Стоимость:** Средняя стоимость оптимизации (необходимо улучшение архитектуры кэширования и индексации в БД).  

**Сложность:** Средняя сложность (могут понадобиться доработки в архитектуре, увеличение мощностей серверов).  

**Зависимости:** Зависит от стабильности работы серверной инфраструктуры и времени ответа базы данных.  

**Конфликт:** Невозможность улучшить производительность без увеличения затрат на оборудование и инфраструктуру.  

**Компромисс:** Если система будет слишком сильно оптимизирована под скорость, может пострадать её гибкость или без-опасность (например, менее строгие проверки входных данных).  

**История:** v1.0 — начальная версия.
